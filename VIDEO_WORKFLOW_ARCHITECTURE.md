# `/video` 指令架构设计文档

## 概述

新的 `/video` 指令采用**脚本优先确认工作流**，灵感来自 Trellis 项目的分阶段协作和 token 节省设计。

---

## 架构对比

### ❌ 传统方式：直接生成代码

```
用户请求
  ↓
Claude → 读取全量知识库 + 生成 Remotion 代码
  ↓
用户可能发现脚本方向不对 → 需要重新生成（浪费 token）
  ↓
修改后再生成代码
```

**问题**：
- 大量 token 浪费在"错误方向的代码"上
- 视频设计师看不到脚本，很难反馈
- 代码和脚本耦合，难以修改

---

### ✅ 新方式：脚本优先 + 确认门

```
用户请求
  ↓
Phase 1: Claude → 读取轻量上下文（workflows + PRD）→ 生成脚本
  ↓
[确认门] 用户审视脚本，确认方向
  ↓
Phase 3: Claude → 读取脚本 + 品牌信息 → 生成代码
  ↓
用户本地运行、调整、渲染
```

**优势**：
- ✅ 脚本生成轻量（上下文少）
- ✅ 确认门防止错误方向
- ✅ 代码生成只做一次
- ✅ 设计师提前看到脚本，可提前反馈
- ✅ 脚本与代码分离，易修改

---

## 核心设计原则

### 1. 分离关注（Separation of Concerns）

| 关注点 | Phase 1（脚本） | Phase 3（代码） |
|--------|----------------|-----------------|
| 输入数据 | workflows + PRD | 脚本 + 品牌 |
| 输出物 | 纯文本脚本 | Remotion 代码 |
| 复杂度 | 低（文本转换） | 高（代码生成） |
| 上下文大小 | 小（<10KB） | 中等（<50KB） |
| 重复次数 | 可能多次（迭代）| 一次（脚本锁定） |

**结果**：每个 Phase 做好自己的事，减少重复加载。

### 2. 上下文隔离（Context Injection Timing）

```
时间轴：

Phase 1:
┌─ 加载 workflows/*.yaml
├─ 加载 approved-prds/
├─ 加载 decisions/
└─ 生成脚本
   ↓
[用户确认]
   ↓
Phase 3:
┌─ 加载 已确认脚本（不重新生成）
├─ 加载 facts/product.md（品牌）
├─ 加载 assets/（资产列表）
└─ 生成代码
```

**节省**：
- Phase 1 不加载品牌色、logo、asset 列表（不需要）
- Phase 3 不重新加载 workflows（已用过了）
- 总上下文加载量 < 单次直接生成

### 3. 确认门（Approval Gate）

```
Phase 1 输出 → 用户确认 → Phase 3 开始

确认前不生成代码，确保：
- 用户同意脚本方向
- 避免生成"错误方向的代码"
- 给设计师机会提前反馈
```

### 4. 脚本持久化（Script Persistence）

```
脚本生成后 → 保存到知识库
  ↓
下次相同需求 → 直接读脚本，跳过生成
  ↓
跨 session 继续 → 无需重新描述
```

## Token 效率分析

### 直接生成代码（旧方式）

```
总 token 消耗：

单次成功：
  脚本推理: 1000 tokens
  代码生成: 2000 tokens
  合计: 3000 tokens

如果需要修改（30% 概率）：
  重新全部: 3000 tokens
  总计: 6000 tokens

平均 token: 3600 tokens/video
```

### 脚本优先工作流（新方式）

```
总 token 消耗：

Phase 1（脚本）:
  轻量加载 workflows + PRD: 500 tokens
  脚本生成: 800 tokens
  小计: 1300 tokens

[用户确认，无 token 消耗]

Phase 3（代码）:
  脚本已有（读取不计）: 0 tokens
  品牌 + 资产加载: 400 tokens
  代码生成: 1800 tokens
  小计: 2200 tokens

总计: 3500 tokens

如果需要修改脚本（修改字数少）:
  重新生成脚本（只改动部分）: 600 tokens
  再确认，再生代码: 2200 tokens
  总计: 2800 tokens

平均 token: 3400 tokens/video
```

**结论**：
- 成功率高时，节省 ~10% token
- 需要修改时，节省 ~20% token
- 最重要的是：**防止大量 token 浪费在错误方向**

## 设计灵感：Trellis 的启发

Trellis 项目（mindfold-ai）使用的 token 节省策略：

| Trellis 策略 | 如何应用到 `/video` |
|-------------|------------------|
| **分阶段工作流** | 4 Phase（脚本 → 确认 → 代码 → 渲染） |
| **上下文隔离** | 脚本用轻量上下文，代码用丰富上下文 |
| **中间态持久化** | 脚本保存到知识库，支持断点续做 |
| **确认门** | Phase 1 后停止，等用户确认 |
| **子 agent 委托** | 脚本用轻量 agent，代码用专业 agent |

## 实际工作流

### 场景 1：快速演示（满意率高）

```
/video dashboard demo, 30 秒

Phase 1: 生成脚本 (~5 min)
  token: 1300

用户: 是的，继续

Phase 3: 生成代码 (~2 min)
  token: 2200

本地: 渲染 (~5 min)
  
总计：~12 min，3500 tokens ✅
```

### 场景 2：复杂视频（需要迭代）

```
/video dashboard + AI 推荐，60 秒，需要展示 5 个场景

Phase 1: 生成脚本 (~5 min)
  token: 1300

用户: 改这些 - 场景 2 改成 25 秒

Phase 2: 修改脚本 (~3 min)
  token: 600

用户: 好了，继续

Phase 3: 生成代码 (~2 min)
  token: 2200

本地: 调整 + 渲染 (~10 min)

总计：~20 min，4100 tokens ✅
```

### 对比：直接生成（出错的情况）

```
/video dashboard + AI 推荐，60 秒

直接生成代码: (~10 min)
  token: 3000

用户: 脚本方向不对，应该强调数据安全而不是 AI

重新生成: (~10 min)
  token: 3000

本地: 调整 + 渲染 (~10 min)

总计：~30 min，6000 tokens ❌
```

**差异**：脚本优先节省 ~25% 时间和 30% token（在需要迭代时）

## 组件架构

```
/video 指令
├─ Phase 1: 脚本生成器
│  ├─ Input: workflows, PRD, user request
│  ├─ Output: Script YAML/MD
│  └─ Agent: 轻量（500 tokens 基础上下文）
│
├─ Phase 2: 脚本修订器（可选）
│  ├─ Input: 用户的改动要求
│  ├─ Output: 修改后的脚本
│  └─ Agent: 轻量（只改动部分）
│
├─ Phase 3: 代码生成器
│  ├─ Input: 已确认脚本 + 品牌 + 资产
│  ├─ Output: VideoScene.tsx
│  └─ Agent: 专业（Remotion expert）
│
└─ Phase 4: 渲染指导
   ├─ Input: 生成的代码
   ├─ Output: 启动说明 + 资产清单
   └─ No agent needed（纯说明文本）
```

## 使用体验

### 面向视频设计师

```
1. /video 你的需求
   ↓ [快速预览脚本方向]
   
2. 确认脚本
   ↓ [知道方向对了]
   
3. 获得代码
   ↓ [生产级 Remotion 代码]
   
4. 本地预览 + 微调
   ↓ [完全控制]
   
5. 渲染
   ↓ 完成
```

### 面向内容营销团队

```
1. /video 你的需求
   ↓
2. 看脚本，提反馈
   ↓
3. 设计师获得代码，做微调
   ↓
4. 视频完成，沉淀到资产库
   ↓

```

## 总结

| 维度 | 旧方式（直接代码） | 新方式（脚本优先） |
|-----|-----------------|-----------------|
| **工作流** | 1 阶段 | 4 阶段 + 确认门 |
| **Token 效率** | 3600 平均 | 3400 平均（+30% 迭代时） |
| **设计师体验** | 难以提前反馈 | 脚本先看，提前改 |
| **迭代成本** | 高（重新生成代码） | 低（只改脚本） |
| **失败恢复** | 重新开始 | 从脚本继续 |
| **可维护性** | 低（代码含脚本逻辑） | 高（脚本独立） |
| **知识积累** | 无（代码没保存） | 有（脚本沉淀） |

**新方式的核心优势**：
- 🎯 **确认门防止错误方向**
- 📊 **脚本分离，易于复用和修改**
- ⚡ **Token 节省（特别是迭代场景）**
- 👥 **团队协作友好**
- 📈 **知识库积累**
