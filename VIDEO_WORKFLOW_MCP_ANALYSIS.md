# `/video` 指令：MCP 方案 vs 脚本优先方案对比

## 现状

目前发现的 Remotion MCP：
- **remotion-media-mcp** — 文本转视频 + 图像动画 (Veo 3.1)
- **rodumani** — Remotion editor MCP server

---

## 三种方案对比

### 方案 A：直接调用 Remotion MCP（当前新发现）

**工作流**：
```
用户请求
  ↓
Claude → 调用 remotion-media-mcp 
  ↓
MCP 直接生成视频 MP4（8 秒）
  ↓
输出 video.mp4
```

**优点**：
- ✅ **最简洁**：一步生成视频，无需本地运行
- ✅ **全自动**：无需用户操作
- ✅ **快速**：即时输出

**缺点**：
- ❌ **脚本不可控**：AI 直接生成，用户无法提前审视脚本
- ❌ **时长固定**：Veo 3.1 生成 ~8 秒，难以扩展到 30/60 秒
- ❌ **品牌定制差**：难以精确控制品牌色、logo、文案
- ❌ **Token 浪费**：如果结果方向错，需重新调用（全部重新生成）
- ❌ **知识库离线**：脚本从 AI 生成，不基于交互流程（workflows/）
- ⚠️ **成本高**：每次调用 MCP 都要调用外部 API（Veo、Suno 等）
- ⚠️ **依赖外部服务**：需要 API 密钥，服务可用性风险

**适用场景**：
- 快速演示、社交媒体短视频
- 非常简单的场景（< 8 秒）
- 对品牌精度要求不高

---

### 方案 B：当前方案（脚本优先 + 本地 Remotion）

**工作流**：
```
用户请求
  ↓
Phase 1: Claude → 脚本生成（轻量）
  ↓
[用户确认脚本]
  ↓
Phase 3: Claude → Remotion 代码生成
  ↓
用户本地: npm install → render → MP4
```

**优点**：
- ✅ **脚本优先**：用户提前审视，确认方向
- ✅ **高度定制**：精确控制品牌色、logo、动画、时长
- ✅ **Token 高效**：脚本确认后才生成代码，防止浪费
- ✅ **知识库集成**：脚本基于 workflows/ 和 PRD，有上下文
- ✅ **时长灵活**：支持 15s / 30s / 60s / 自定义
- ✅ **品牌一致**：自动读取 facts/product.md 的品牌色
- ✅ **可迭代**：脚本修改轻量，代码一次生成
- ✅ **成本低**：完全本地运行，无 API 调用
- ✅ **离线工作**：不依赖外部服务

**缺点**：
- ⚠️ **需要本地运行**：用户要安装 Node.js + Remotion
- ⚠️ **需要用户确认**：不是全自动（设计上的优势，实际上是"步骤多"）
- ⚠️ **渲染耗时**：30s 视频本地渲染需 2-3 分钟

**适用场景**：
- 产品演示、功能教程（15-60s）
- 内容营销视频
- 需要品牌精度的场景
- 需要迭代调整的视频

---

### 方案 C：混合方案（脚本优先 + 可选 MCP 快速渲染）

**工作流**：
```
用户请求
  ↓
Phase 1: Claude → 脚本生成
  ↓
[用户确认脚本]
  ↓
[用户选择渲染方式]
  ├─ 选项 1：本地渲染（当前方案）
  │  └─ Phase 3: 生成 Remotion 代码 → 本地 npm render
  │
  └─ 选项 2：MCP 快速渲染（新增）
     └─ 调用 remotion-media-mcp → 秒速出视频（8s）
```

**特点**：
- ✅ 结合两者优点
- ✅ 脚本仍然优先确认（高质量保障）
- ✅ 可选快速渲染（低成本、秒速）
- ✅ 可选精细渲染（高品牌精度、全控制）

**实现复杂度**：中等

## Token 消耗对比

### 场景：制作 30 秒产品演示

**MCP 直接调用**：
```
用户: /video dashboard 演示

Claude: 
  思考脚本 + 调用 MCP: 1500 tokens
  
MCP 返回 8 秒视频 (不是 30 秒，需要多个片段)

如果结果不对，重新调用 MCP: 再 1500 tokens

平均: 1500-3000 tokens
额外成本: MCP API 调用费用
```

**脚本优先（当前）**：
```
Phase 1: 脚本生成: 1300 tokens
[用户确认]
Phase 3: 代码生成: 2200 tokens

平均: 3500 tokens
额外成本: 0（本地运行）
```

**混合方案**：
```
Phase 1: 脚本生成: 1300 tokens
[用户确认]
[选择快速渲染]
MCP 调用: 1000 tokens + API 费用
  或
[选择精细渲染]
Phase 3: 代码生成: 2200 tokens

平均: 2300-3500 tokens（取决于选择）
额外成本: 可选
```

## 建议

### 短期（推荐当前方案）

**采用**：脚本优先 + 本地 Remotion

**原因**：
- 脚本审视是核心价值（防止浪费）
- 品牌精度对产品视频至关重要
- 完全控制 + 高可迭代性
- 零成本（无 API 调用）
- 知识库集成深度

### 中期（可考虑混合方案）

**扩展**：添加 `/video --fast` 选项，使用 MCP 快速渲染

**流程**：
```
/video --fast [主题] 
  ↓ [秒速生成 8s 视频，用于快速验证]

/video [主题]
  ↓ [标准流程：脚本优先，本地精细渲染]
```

**使用场景**：
- `--fast`：社交媒体短视频、快速演示、验证概念
- 标准：营销视频、教程、正式发布

### 长期（技术预研）

- 监测 Remotion MCP 的演进（目前还在早期）
- 评估 Veo 3.1 的品质和定制能力
- 考虑使用 MCP 作为"渲染后端"（而不是替代脚本阶段）

---

## 关键结论

**MCP 直接生成 ≠ 节省 Token**

反而可能因为：
1. ❌ 脚本不可控 → 结果错误 → 重新调用 → Token 浪费
2. ❌ 时长固定 → 不够 30s → 多次调用 → 更多 Token
3. ❌ 品牌难定制 → 结果不满意 → 重新调用 → Token 浪费

**脚本优先 = 最佳 Token 效率**

因为：
1. ✅ 脚本确认 → 方向正确 → 代码一次生成
2. ✅ 支持完整时长 → 一次渲染完成
3. ✅ 知识库集成 → 品牌精度内置
4. ✅ 迭代时只改脚本（轻量）→ 代码不变
