# Product Design Rules

## 用户身份的多样性

设计面向「受害者」的功能时，要考虑实际填写人可能不是受害者本人。你的平台上还有调查人员、专业分析师等用户，可能受受害者委托代为操作。

文案要保持中性视角:

- 字段命名避免假设第一人称: "Your Account" → "Victim / Affected Account"
- placeholder/hint 用第三人称或被动语态: "the affected account" 而非 "your account"
- 引导文案不直接说 "you 怎么样了": "If funds were lost" 而非 "If you've lost funds"
- 共情语气换成对填写动作的感谢: "Thank you for taking the time to report" 而非 "We're sorry this happened to you"
- 必要时用 "the report" 替代 "your report"

中性 ≠ 冷漠。可以保留温度，但温度对准的是「举报这个动作的价值」，而不是「填写者的处境」。

## 字段命名: 直白 vs 中性

法律中性的命名（如 "Suspected Account"）适合通用语境。但如果页面上下文已经明确是滥用举报（标题就是 "Report Abuse"），用更直白的 "Offending Account" 反而更易懂。

判断标准:
- 页面标题是否已经框定了语境 → 是，可以直白
- 是否有 disclaimer 兜底说明 → 有，可以直白
- 后台逻辑是否仍然把用户提交视为 "suspected" 待验证 → 是，前台直白不影响后端严谨

## Metadata 跟项绑定，不放表单层级

当一个 metadata 字段（如 类别、单位、地区、币种 decimals）天然属于某个 item 而非整个表单时，要把它内嵌到 item 行内，不要做成全表单一个公共字段。

判断标准: 「同一表单里不同 item 的这个字段值会不会不一样？」如果答案是「会」，就该跟 item 绑定。

示例:
- 类别与条目绑定: 一个工单可能涉及多个不同类别的条目，不应做成表单顶层 类别 选择器
- 币种与金额绑定: 多笔交易可能不同币种
- 时区与时间戳绑定: 多个时间事件可能跨时区

代价: UI 复杂度上升、字段宽度受限。可在批量添加（Bulk add）面板顶部放一个共享 metadata 选择器，应用到本批所有 item，平衡输入效率。

## 多输入字段 (Multi-Input)

当字段值天然可能是多个时（多个受影响账户、多个相关链接、多个联系邮箱），不要强行用单一输入框或纯 textarea：

- 默认显示一个输入框
- 输入框右侧 × 按钮可移除（仅多于一行时显示）
- 下方 "+ Add another" 按钮新增一行
- 如果用户可能粘贴大批量（如运营人员），额外提供 "Bulk add"，弹出 textarea 批量粘贴后切分成多行

为什么不用纯 textarea「一行一个」: 单独输入框可以独立校验、独立删除，更结构化；textarea 适合用户已经有现成数据需要快速粘贴的场景，作为补充手段。

删除按钮的统一行为: 任何行点击 × 都直接移除该行；删除后如果总行数变成 0，再补一个空行。**不要**为「第一行」单独写「删除后立即补一个」的特殊逻辑——这会导致用户永远删不掉第一行，看起来像 bug。

## 默认值要慎用「逃逸选项」

下拉选择字段不要把「跳过/不知道/我不确定」类选项设为默认值。这类选项默认勾选会让多数用户懒得思考直接通过，数据质量会暴跌。

正确做法:
- 默认无选中 + 必填校验，强制用户主动选择
- 把「不知道」类选项保留在列表中（通常放末尾），让真分不清的用户能选
- 提示文案说明为什么需要这个分类，降低抗拒

判断标准: 你希望多数用户停下来思考一下再选，还是只关心提交本身。如果是前者，不要给逃逸出口当默认。

## 字段顺序与渐进展开

如果表单中有「触发性」字段（填完它之后会自动展开/弹出新区域），把这类字段排在所有需要展示的选填字段之后。否则触发后的视觉变化会从用户视线里抢走注意力，让前面的选填字段被跳过。

具体做法: 在排序时把这类必填字段放在「最后一个必填」位置，前面塞需要展示的选填字段。这样用户填完触发性字段时，已经路过所有选填字段。

判断哪些字段是「触发性」: 填完后会引发明显视觉变化（区域展开、跳到下一步、显示新内容），就是触发性的。

## 前端格式校验

涉及结构化输入的字段（账户 ID、邮箱、URL、电话等），前端必须在用户输入时做格式校验，避免后端收到无效数据：

- blur 时触发，不实时打断输入
- 错误提示就近显示在该输入下方，明确指出问题（"Invalid email format" 而非 "Invalid input"）
- 多格式场景下，根据相关上下文（如选定类型）切换校验规则
- 仅校验格式（语法层面），不做存在性校验（如账户是否真实存在，留给后端）
- 未提供必要上下文时不校验（如未选类型就不校验格式）

为什么不全靠后端: 用户填错时立刻能看到反馈，而不是提交后才发现，省一次往返；后端仍然要校验，前端只是第一道防线。

## 二选一输入: URL 或 文件

当一项数据可以通过 URL 或文件两种形式提供（如 attachment 可以是链接也可以是截图），把两个输入并列在同一行，中间用 "OR" 标识，让用户自由选择：

```
[URL input] OR [Drop file or browse] [×]
```

每行的两个输入都是 optional，提交时空行忽略；只要任一非空就视为一项有效证据。比强制要求统一格式（只能上传文件 / 只能填 URL）灵活很多，覆盖更多用户场景。

## 「至少填一个」字段

某些字段不是单独必填，而是在一组里至少填一个（例: Offending Account 和 Offending Website 至少填一个）：

- 不打红色 *，改用 (at least one of X or Y) 的淡色提示
- 提交时若整组都为空，整组都标红，错误信息说明「至少提供一个」
- 不要在两个字段下方分别写错误提示（重复且 noisy），写在其中一个或独立位置即可
