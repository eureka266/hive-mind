# PRD Writing Rules

## 语言

- 内容尽量用中文写
- 平台文案（实际显示给用户的内容，如 placeholder、hint、按钮文案、错误提示）保留英文
- 字段名、enum 值、API 字段、技术专有名词保留英文
- 你自己产品的专有术语必须保留英文，包括产品名、功能名、API 字段名、模块名和竞品名等既有产品术语
- 描述性英文词（如 crowdsourcing、segmentation、onboarding）默认改用中文表达，避免读者理解负担；但如果属于你产品的、API 的、字段名或既有产品术语，则按上一条保留英文
- 表格中的说明性列（类型、必填、说明、备注等）用中文，不要写 "Yes/No/Select/Required"
- 不强制翻译已有英文术语，保持自然
- `customer-knowledge/` 下的所有知识库内容必须使用英文，包括 user manual、FAQ、API doc、示例、标题和 metadata 描述；聊天解释仍可用中文

## 格式

- 不写 PRD 大标题（如 "# PRD: XXX"）。文件会导入文档工具，文档工具会用文件名作为标题。所有章节标题从 H1 开始
- 不使用水平分割线（`---`）。用空行和标题层级划分章节
- 一级标题尽量少，相关内容合并到二三级。常见合并：
  - 「背景与目标」: 概述、背景、目标、非目标合到一节，目标和非目标用粗体小标题区分
  - 「数据」: schema、映射、流程合到一节
  - 「视觉与文案」: 风格、文案原则、accessibility 合到一节

## 内容

- PRD 是内部文档，目标和价值用内部视角写（如"扩充地址标签库"），不照搬面向用户的宣传话术（如"同步至合作的交易所"）
- 内容简洁，避免冗余说明
- 范围说明放在概述里用 blockquote 标注，不专门开章节
- 设计决策记录 why，用 blockquote 标注。但要克制使用：
  - 只记录读者可能产生疑问的决策（如「为什么不用 X 方案」）
  - 字段标注 optional/required 已经表达的、看一眼就懂的，不需要再写决策说明
  - 同一主题的多条决策合并成一条
  - 措辞精简，能一句话说清不写两句

## 章节结构与层级

PRD 要有清晰的逻辑分层。一般组织方式:

1. 原型（H1）
2. 背景与目标（H1）
3. 方案（H1）
   - 各模块改动（H2）
     - 模块内的子区域（H3，必要时 H4）
   - 状态设计 / 通用交互模式等横向章节（H2）
4. 权限差异 / 视觉规范 等独立维度（H1，按需）

让"方案"作为容器把所有模块改动收纳起来，避免每个模块改动都顶到 H1 让目录扁平、读者不知道彼此的层级关系。

## PM 范围 vs UI/UX 范围

PRD 是 PM 写的，描述**业务逻辑、交互行为、文案**。**视觉样式细节**（颜色 hex、字号、间距、阴影、圆角等）属于 UI/UX 范围，不写进 PRD。

PM 该写:
- 交互逻辑（hover / click / 选中 / 禁用等状态下发生什么）
- 文案（按钮文字、placeholder、tooltip 内容、错误提示等）
- 字段必有 / 可选、字段适用条件
- 数据展示规则（排序、筛选、分组、上限等）
- 状态分支（如免费用户 vs 付费用户、首次使用 vs 再次使用）

PM 不该写:
- 具体颜色 hex（除非是定义全局色板/token 映射这种规范性内容）
- 字号、字重、行高
- padding、margin、border-radius、shadow
- 具体动画 timing / easing

## 描述要具体到能直接实现

涉及交互、文案、状态的描述要落到可执行的细节，避免模糊词:

- 不写"显示提示"，写"hover tooltip 文案: 'xxx'"
- 不写"禁用按钮"，写"button 置灰且不响应 click，可选 tooltip 文案: 'xxx'"
- 不写"高亮"，写"行左侧出现一条紫色高亮条作为选中标识"（描述视觉信号，不指定具体颜色值）
- 不写"特殊处理"，写出具体的特殊情况和分支逻辑

判断标准: 开发/测试只看 PRD 应该能照着做，不需要回头问"具体什么文案 / 什么行为 / 什么状态"。视觉细节等 UI/UX 出。

## 字段表格组织

PRD 描述模块/组件结构时常用表格列字段。两条经验:

- **同类项合并行**: 如果多个字段共用相同规则，合并为一行（如 "导入成功 / 导入失败 / 部分失败 状态标签 | 系统识别 | 无 | Source"），不要每个类型单独列一行
- **单字段承载多语义则拆分行**: 如果一个 UI 元素同时表达多种信息（如徽章上的 "类型" + "具体名称"），拆为多个独立行（如 "Status Type" 和 "Detail"），让每行只描述一个语义维度

字段表格必带的列:

- **必有 / 可选**: 明确指出哪些是必展示的字段，哪些按数据存在与否展示。这是开发实现的核心信息

## PRD 写什么 vs 不写什么

**优化型 PRD 只写改动部分**: 如果 PRD 是对已有功能的优化（如本次详情页重设计），不要把整个页面/功能从头到尾描述一遍。只写相关有变动的部分，未改动的现有逻辑不重复描述。读者默认知道现状。

**不要把对话内容写进 PRD**: 沟通过程中讨论的背景、问题分析、备选方案对比、推导过程，不属于 PRD 内容。PRD 只写最终的设计决策和实施细节。需要保留 why 的，用 blockquote 一句话说清即可。

**不要描述全局复用元素**: 如果某区域是全局复用、本次不改动（如 sidebar、全局 header），不要写进 PRD。即使提一句 "不在本次范围"，也是冗余。直接跳过。

**判断标准**: 删掉这段话，开发/设计能否准确实现剩下的内容？如果能，就该删。
