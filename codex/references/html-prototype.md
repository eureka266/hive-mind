# HTML Prototype Workflow

Use this when the user wants a clickable prototype that is fast to review.

## Default Constraints

- Single HTML file.
- Inline CSS.
- Vanilla JavaScript.
- No framework or bundler unless requested.
- Realistic copy, not lorem ipsum.
- Include meaningful states when they affect the decision.

## Include These States When Relevant

- Empty state.
- Loading state.
- Populated state.
- Error state.
- Permission or compliance warning state.

## Prototype Structure

1. Entry point: one obvious starting screen or dashboard.
2. Primary task path: the one flow that proves the product idea.
3. Unhappy path: one failure, risk, permission, or compliance edge case that would change the design.
4. State notes: what is mocked and what represents intended behavior.

## Review Checks

- Does the user know what to do first?
- Does the interface expose key status and risk information clearly?
- Are the controls aligned with the user permission level?
- Is at least one important failure case represented?
- Is anything visually prominent but product-wise unimportant?

## Handoff Note

When returning the prototype, summarize the main path, major states, and intentional out-of-scope items.
