# 真实使用场景

> **核心机制，一句话说明。**
>
> 你输入 `/prd 批量导入`。在提第一个问题之前，HiveMind 会先把知识库里的相关内容全部加载进来——产品事实（facts/）、涉及类似范围的历史决策（decisions/）、可复用的产品规则（memory/rules/）。只有加载完成之后，它才开始追问——每次只问一个问题，绝不说"很有意思！"，绝不跳过没答清楚的问题。
>
> 下面每个场景都在演示这个模式。

---

**追问逻辑**参考了 [gstack](https://github.com/garrytan/gstack) 的 `/office-hours`——要求行为证据、点名具体的人、对每个答案表明立场，拒绝模糊回答。`/prd review` 的逻辑参考了 gstack 的 `/plan-ceo-review`——前提挑战、范围风险、以及"六个月后你会后悔什么"这类反向追问，用在产品决策上。

## 1. 设计转化升级引导流程

**适用时机：** 付费功能有转化潜力，但免费用户从未看到有吸引力的升级提示——你需要设计分层展示逻辑，同时不打扰新用户。

### 1.1 场景描述

你的 SaaS 有一个监控/告警功能（比如"某指标变化时立即通知我"），该功能仅限订阅用户使用。免费用户可能无意中触发它（比如悬停在某个按钮上），但随即遇到一道没有任何上下文的付费墙。你发现有一批用户反复查看同一条记录——这是他们想持续追踪的强烈信号，但他们不知道产品提供了这个能力。

PM 需要设计三层递进式升级引导。

### 1.2 起始提示词

```
/prd 监控功能升级引导

7 天内重复查看同一记录 2 次以上的用户，是我们最强的升级信号。
目前这些用户直接撞上硬付费墙。我想设计三层递进式升级引导。
```

### 1.3 第一步：加载知识库

在追问任何内容之前，HiveMind 先加载：

- `facts/[产品].md` → 当前套餐和哪个套餐包含监控功能
- `decisions/` → 关于这个功能的历史决策（比如为什么是硬付费墙而非软提示）
- `memory/rules/` → 关于转化文案或抑制窗口的可复用规则

加载完成后才开始追问——因为已经知道哪些问题已经有了答案，它不会重新打开已经关闭的讨论。

### 1.4 追问过程（每次只问一个）

```
AI："在设计分层之前——我们的目标是免费用户、按量付费用户，还是两者都有？
    现有决策里没有区分这两类用户。如果你说'都有'，
    我会标出两类用户的文案在哪里需要分叉。"

PM："都有，但按量付费用户需要不同的文案。"

AI："明白。用户关闭提示后的抑制窗口是多久？
    我看到 decision-20250312-quota-exhaustion 里对类似的引导用了 30 天。
    这里是否沿用 30 天，还是设一个不同的时长？"

PM："30 天就好。"

AI："最后一个问题：第二层（重复访问触发）——
    '重复'是指同一用户在 7 天内查看同一条记录 2 次以上？
    还是任意 2 次查看任意记录都算？
    这两种定义对后端计数逻辑影响很大。"
```

当 PM 已经明确到足以写出无歧义 PRD 的程度，追问就停止。知识库里能回答的问题，HiveMind 不会再问你。

### 1.5 待提交内容

讨论结束后，HiveMind 将以下内容暂存，等你确认后再写入：

```
✅ 等待你确认后提交：

  features/monitor-upsell/prd.md          （新建——触发逻辑表、文案变体、埋点事件清单）
  features/monitor-upsell/workflow.yaml   （新建——三层状态机）
  features/monitor-upsell/prototype.html  （新建——展示三层交互的可点击原型）
  decisions/decision-YYYYMMDD-monitor-upsell.md  （新建——记录为什么选 7 天窗口）
  memory/rules/conversion-suppression.md  （新建——"30 天关闭抑制规则"）
```

你不批准，什么都不写。

### 1.6 完成标准

- 原型完整演示三层交互跳转
- 关闭抑制逻辑已文档化，包含边界情况
- 研发交接文档包含 ≥ 8 个覆盖完整转化漏斗的埋点事件
- "为什么选 7 天"的原因在 `decisions/` 里，不是只存在 Slack 消息里

## 2. 重构产品定价模型

**适用时机：** 需要重构套餐（新套餐名称、配额调整、老用户迁移），且涉及多个子 PRD 需要同步交付、互不矛盾。

### 2.1 场景描述

你的产品已经超越了最初的三级定价模型。团队决定迁移到新结构，并将一个高价值功能（监控/告警）拆成独立付费附加项。这次改版涉及 6 个相互关联的子 PRD：Pricing 页、Checkout、Billing、配额执行、老用户迁移、邮件触发。

### 2.2 起始提示词

```
/prd 定价模型 V3

我们要上线重新设计的定价模型，监控功能单独作为付费附加项。
需要子 PRD：Pricing 页、Checkout、Billing、配额执行、老用户迁移、
邮件触发（配额预警和套餐变更）。

核心约束：老用户在明确同意之前，不能面临任何价格上涨。
```

### 2.3 第一步：加载知识库

HiveMind 先加载：

- `facts/[产品].md` → 当前套餐名称、配额数字、精确定价
- `decisions/` → 关于降级逻辑、Top-up 规则、账单周期的历史决策
- `memory/rules/product-facts-pricing.md` → 已有的定价权威规则

你说的任何内容只要和 `facts/` 里的数据有矛盾，就会被标记为 `[CONFLICT]`，在 PRD 动笔前就提示你。

### 2.4 追问过程（每次只问一个）

```
AI："现有 facts 显示历史 Growth 套餐用户每月 500 次配额。
    你的新 Essential 套餐是 300 次。
    你说'不涨价就不能动'——这意味着历史用户保留 500 次额度，
    还是以旧价格锁定在新的 300 次上限上？
    这是两条完全不同的迁移路径。"

PM："保留配额，旧价格锁定 6 个月，然后弹出引导让他们选择。"

AI："明白。这 6 个月的过渡期是从上线日算起对所有用户统一生效，
    还是以每位用户的下一个续费日为起点？
    Billing 页的展示逻辑会因此不同。"

PM："以续费日为起点。"

AI："最后一个：监控附加项在所有套餐上都可以购买，
    还是只对付费套餐开放？
    这会影响 Pricing 页的功能对比表格。"
```

### 2.5 待提交内容

```
✅ 等待你确认后提交：

  features/pricing-v3/prd.md                      （新建——主文档，链接所有子 PRD）
  features/pricing-v3/sub-prds/pricing-page.md    （新建）
  features/pricing-v3/sub-prds/checkout.md        （新建）
  features/pricing-v3/sub-prds/billing.md         （新建）
  features/pricing-v3/sub-prds/quota.md           （新建）
  features/pricing-v3/sub-prds/migration.md       （新建——映射表、过渡期规则）
  features/pricing-v3/sub-prds/emails.md          （新建——三类邮件触发场景）
  decisions/decision-YYYYMMDD-pricing-v3-migration.md  （新建）
  facts/[产品].md                                  （更新——新套餐名称和配额）
```

### 2.6 完成标准

- 所有子 PRD 交叉引用一致（相同套餐名称、相同配额数字）
- 迁移表覆盖 100% 的历史套餐类型
- `facts/[产品].md` 已更新——它现在是新定价的权威来源

## 3. 构建嵌入式 AI 助手知识库

**适用时机：** 你的产品有（或计划有）一个产品内置 AI 客服，需要 RAG 知识库——并且随着产品迭代，这个知识库需要保持准确和最新。

### 3.1 场景描述

你的 SaaS 有一个嵌入式 AI 助手，能在产品内回答用户问题。知识库是上线时一次性写的，现在已经三个月没更新了——定价过时、引用了已下线的功能、与当前帮助文档相互矛盾。PM 需要系统性地重建并维护它。

### 3.2 起始提示词

```
/prd AI 助手知识库

我们的嵌入式 AI 知识库已经三个月没更新了，需要系统性重建。
模块：user_manual/、pricing/、faq/（约 40 条来自真实工单的 Q&A）、api_doc/。

硬性规则：不能写未上线的功能、不能写内部实现细节、
每条声明都必须能追溯到 facts/ 或已批准的 PRD。
```

### 3.3 第一步：加载知识库

HiveMind 先加载：

- `facts/[产品].md` → 当前已上线、已确认的功能清单
- `approved-prds/` → 任何具体行为声明的来源文档
- `memory/rules/rag-kb-authoring.md` → 如果存在，则是团队已建立的 KB 撰写规范

任何无法追溯到上述来源的内容，它都不会生成。

### 3.4 追问过程（每次只问一个）

```
AI："现有 facts 文件里没有指定数据导出的字段数量。
    我在不同历史 PRD 里看到了两个数字：13 个字段和 22 个字段。
    哪个是当前版本？在这个问题确认之前，我不会写导出指引。"

PM："22 个字段，这是当前版本。"

AI："明白。FAQ 模块：你是希望我从你粘贴的支持工单里提取 Q&A 对，
    还是基于现有 facts 文件来起草 FAQ？
    前者更贴近用户真实困惑，后者更快但可能错过用户实际会问的问题。"

PM："从工单里提取——我现在粘贴 20 条样本。"
```

### 3.5 待提交内容

```
✅ 等待你确认后提交：

  ai-kb/user_manual/feature-guide.md      （新建——分步指引，用户视角）
  ai-kb/pricing/plans-and-billing.md      （新建——仅当前套餐，不含定价策略）
  ai-kb/faq/common-faq.md                （新建——40 条从工单提取的 Q&A）
  ai-kb/api_doc/reference.md             （新建——REST/Webhook/鉴权）
  memory/rules/rag-kb-authoring.md       （新建——"每个 H2 = 独立检索单元"）
  decisions/decision-YYYYMMDD-ai-kb-scope.md  （新建——哪些内容被排除及原因）
```

### 3.6 完成标准

- 每一条内容都能追溯到权威来源（`facts/` 或已批准的 PRD）
- 撰写规范已保存到 `memory/rules/`，确保后续更新遵循同一标准
- AI 助手能正确回答一组 10 个测试问题，不出现幻觉

## 4. 追踪跨功能产品决策

**适用时机：** 多个功能共享同一底层策略（配额规则、权限、定价行为），但因为最初的决策原因没有记录下来，同一个问题被反复讨论。

### 4.1 场景描述

过去一年，你的 PM 团队交付了 6 个功能，其中 3 个涉及配额执行。但没人记得：免费套餐为什么是每月 10 次查询（不是 5 次或 20 次）；配额耗尽为什么是硬封锁而非软提示；降级为什么等到账单周期结束才生效。这些问题每个迭代都会被重新问到。

### 4.2 起始提示词

```
/prd 决策审计——配额与权限系统

对以下 4 个问题做正式决策日志，记录决定内容、原因、假设条件和边界情况：
1. 免费套餐配额：为什么是每月 10 次，而不是其他数字
2. 配额耗尽时的硬封锁 vs 软提示 + 自动升级
3. 降级生效时间：立即生效 vs 账单周期结束
4. 权限绑定：为什么功能访问权限绑定配额余额，而不只是套餐等级

讨论每个问题之前，先检索 decisions/ 中已有的相关记录。
```

### 4.3 第一步：加载知识库

HiveMind 扫描 `decisions/` 中所有涉及配额、权限或套餐逻辑的条目，然后直接告诉你：

```
AI："找到 2 条相关历史决策：
    - decision-20250312-quota-exhaustion.md：硬封锁是出于合规原因选定的。
      这已经回答了问题 2——你是要重新讨论，还是直接确认并补充文档？
    - decision-20250601-downgrade-policy.md：账单周期结束生效是为了避免折算复杂度。
      这已经回答了问题 3。

    问题 1 和问题 4 没有历史决策记录。从问题 1 开始。"
```

### 4.4 追问过程（每次只问一个）

```
AI："免费套餐 10 次/月这个数字——当时是基于成本模型、
    竞品对标，还是经验判断？
    我这样问是因为：如果是成本导向的，决策日志里需要记录那个假设，
    这样才知道什么时候应该重新评估。"

PM："主要是经验判断——我们想要一个对用户够宽松、但又能推动转化的数字。"

AI："那我会这样记录假设：
    '10 次/月是转化优化的经验值，不是成本推导的下限。
    建议在 6 个月后结合激活率和转化率数据重新验证。'
    这样描述准确吗？"
```

### 4.5 待提交内容

```
✅ 等待你确认后提交：

  decisions/decision-YYYYMMDD-free-quota-10.md             （新建）
  decisions/decision-YYYYMMDD-feature-permission-binding.md  （新建）
  memory/rules/quota-policy.md                             （新建——供未来 PRD 自动加载的提炼规则）
```

### 4.6 完成标准

- 每个问题都有"为什么"的书面记录，不只是"是什么"的答案
- 相关决策之间已建立交叉引用
- 下一位 PM 在 30 秒内能找到"为什么是每月 10 次"的答案

## 5. 数据驱动的流失原因诊断

**适用时机：** 发现某项指标下降（注册、激活、留存），需要综合多个数据源——埋点数据、支持工单、用户访谈——整理成有条理的假设和下一步行动。

### 5.1 场景描述

你注意到新免费用户 7 日留存率出现明显环比下降。手头有：埋点漏斗数据（从"首次操作"到"第 3 天回访"流失率 60%）、标记为"看不懂"的支持工单样本、以及 5 位流失用户的访谈记录。

### 5.2 起始提示词

```
/prd research 三月新用户流失原因分析

诊断新免费用户为什么在第一周流失。结合埋点数据、用户反馈和支持单据分析原因。

可用数据：
1. 埋点漏斗：从"首次操作"到"第 3 天回访"流失率 60%（我来粘贴）
2. 支持工单：20 条标记为"看不懂"的样本（我来粘贴）
3. 访谈记录：上个月 5 位流失用户（我来粘贴）

目标：列出 3 个主要根因并排名，选出优先修复项，写一份可以分享给团队的研究总结。
```

### 5.3 第一步：加载知识库

HiveMind 先检索 `memory/research/` 里的历史研究记录和任何涉及引导或激活的决策，然后直接告诉你：

```
AI："找到一条历史研究记录：memory/research/activation-drop-q3.md。
    它把'第一天空状态'列为假设，但标注为未验证。
    如果你的新数据能印证或推翻它，我们可以把它关闭。
    先粘贴埋点数据，从漏斗开始。"
```

### 5.4 追问过程（每次只问一个）

```
AI："漏斗显示从'首次操作'到'第 3 天回访'流失 60%。
    '首次操作'是指完成一次完整任务，还是只是开始了任务？
    如果用户是开始了但没完成，和完成了但没回来，
    这是两个完全不同的问题。"

PM："是开始了但没完成——大多数用户没有完成第一次任务。"

AI："这改变了假设方向。看你粘贴的支持工单，
    20 条里有 7 条提到'看不懂结果'，而不是'不会操作'。
    这个规律指向结果解读问题，不是任务流程问题。
    你有数据显示是哪类结果最容易让用户困惑吗？"
```

### 5.5 待提交内容

```
✅ 等待你确认后提交：

  memory/research/churn-new-users-march.md              （新建——假设排名、证据来源、置信度）
  decisions/decision-YYYYMMDD-churn-fix-priority.md     （新建——为什么优先修复假设一而非假设二）
```

### 5.6 完成标准

- 3 个假设各自有来自至少 2 个独立数据源的证据支撑
- 选定的修复方向有可证伪的成功指标（"3 周内 7 日留存率提升 X%"）
- 研究总结不超过 1 页——没有参与讨论的 PM 也能直接读懂
