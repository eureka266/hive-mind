## 1. 设计转化升级引导流程

**适用时机：** 付费功能有转化潜力，但免费用户从未看到有吸引力的升级提示——你需要设计分层展示逻辑，同时不打扰新用户。

### 1.1 场景描述

你的 SaaS 产品有一个监控/告警功能（比如"某指标变化时立即通知我"），该功能仅限订阅用户使用。免费用户可能无意中触发它（比如悬停在某个按钮上），但随即遇到一道毫无上下文的付费墙。你发现有一批用户反复查看同一条记录——这是他们想要持续追踪的强烈信号，但他们不知道产品提供了这个能力。

PM 需要设计三层递进式升级引导：

- **第一层**：首次印象（新用户，注册 ≤ 24 小时）：轻量级气泡提示，不强推
- **第二层**：重复行为信号（同一用户在 7 天内重复查看同一记录 2 次以上）：完整升级卡片，展示套餐对比和价格
- **第三层**：明确意图（用户主动点击了付费功能按钮）：升级弹窗，最大化转化

### 1.2 起始提示词

```
/prd 监控功能升级引导

背景：7 天内重复查看同一记录 2 次以上的用户，是我们最强的升级信号。
目前这些用户直接碰到硬付费墙。我需要三层递进式升级引导：
- 第一层（新用户，注册 < 24h）：悬停气泡，轻量提示
- 第二层（重复行为信号）：完整升级卡片，展示套餐对比和价格
- 第三层（主动点击付费功能按钮）：完整升级弹窗，最大化转化

约束条件：
- 用户关闭提示后 30 天内不再展示
- 免费用户和按量付费用户看到不同的文案
- 弹窗不能自动触发，只有用户主动操作才出现
```

### 1.3 工作流

1. **AI** 读取 `facts/[产品].md` → 确认哪些套餐有该功能、哪些没有
2. **AI** 检索 `decisions/` → "我们之前有没有决定过为什么用硬付费墙而非软提示？"
3. **PM + AI** 讨论：每一层的触发逻辑、抑制窗口、面向不同套餐用户的文案变体
4. **AI** 起草 `features/monitor-upsell/prd.md`，包含触发条件表、文案变体、埋点事件清单
5. **PM** 审核——确认或调整抑制窗口和文案
6. **AI** 生成 `features/monitor-upsell/workflow.yaml`——三层状态机
7. **AI** 生成 `features/monitor-upsell/monitor-upsell-prototype.html`——展示三层交互的可点击原型
8. **AI** 写入 `decisions/decision-YYYYMMDD-monitor-upsell.md`——记录"为什么选 7 天窗口、为什么 24 小时是新用户阈值"
9. **PM** 确认 → `git commit`

### 1.4 知识库沉淀内容

- `decisions/` — 分层触发逻辑与阈值（下一位 PM 不需要重新讨论"第二层应该触发 2 次还是 3 次"）
- `memory/rules/` — "重复访问是有效的转化信号；不要与误操作混淆"
- `features/monitor-upsell/` — 完整 PRD、状态机、原型、待确认问题

### 1.5 完成标准

- 原型完整演示三层交互跳转
- 关闭抑制逻辑已文档化，包含边界情况
- 研发交接文档包含 ≥ 8 个覆盖完整转化漏斗的埋点事件

## 2. 重构产品定价模型

**适用时机：** 需要重构套餐（新套餐名称、配额调整、老用户迁移），且涉及多个子 PRD 需要同步交付、互不矛盾。

### 2.1 场景描述

你的产品已经超越了最初的三级定价模型（入门版 / 成长版 / 专业版）。团队决定迁移到新结构，更清晰地划分价值层级，并将一个高价值功能（监控/告警）拆成独立付费附加项。这次改版涉及：

- 全新 Pricing 页（新套餐名称、功能对比表格）
- 重新设计的 Checkout 页（处理升级、降级、首次购买三条路径）
- 新 Billing 页（展示当前套餐、用量明细、附加项管理）
- 配额执行逻辑（不在用户不知情的情况下断供）
- 老用户迁移方案（旧套餐到新套餐的映射、过渡期、通知）
- 自动化邮件触发（配额 80% 预警、配额耗尽、套餐变更确认）

### 2.2 起始提示词

```
/prd 定价模型 V3

我们要上线重新设计的定价模型。需要以下子 PRD：
1. Pricing 页 —— 新套餐结构、功能对比表
2. Checkout 流程 —— 升级 / 首次购买 / 降级三条路径
3. Billing 页 —— 当前套餐展示、用量进度条、附加项管理
4. 配额执行 —— 触达上限时的优雅降级
5. 老用户迁移 —— 旧套餐到新套餐的映射、过渡期
6. 邮件触发 —— 配额预警（80%、耗尽）、套餐变更确认

核心约束：老用户在明确同意之前，不能面临任何价格上涨。
```

### 2.3 工作流

1. **AI** 读取 `facts/[产品].md` → 加载当前套餐名称、配额、定价
2. **AI** 检索 `decisions/` → "我们之前有没有决定过降级为什么不支持自助操作？"
3. **PM + AI** 逐一讨论每个子 PRD，从 Pricing 页开始（它是其他所有内容的锚点）
4. 每个子 PRD 中，**AI** 发现与 `facts/` 矛盾时标记 `[CONFLICT]`
5. **AI** 生成 `features/pricing-v3/prd.md`——包含所有子 PRD 链接的主文档
6. **AI** 生成迁移映射表：旧套餐 → 新套餐 → 各用户类型的变化差异
7. **PM** 逐一确认子 PRD
8. **AI** 自动生成三类邮件文案（配额预警、耗尽、套餐变更确认）
9. **AI** 生成 Pricing 页和 Checkout 流程的原型 HTML
10. **AI** 提交，并记录迁移规则的决策日志

### 2.4 知识库沉淀内容

- `facts/[产品].md` 更新——新套餐名称和配额成为权威来源
- `decisions/` — 老用户迁移规则、降级为何需要联系支持、附加项为何单独销售
- `features/pricing-v3/` — 全部 6 个子 PRD、原型、邮件文案、研发资产

### 2.5 完成标准

- 所有子 PRD 交叉引用一致（相同套餐名称、相同配额数字）
- 迁移表覆盖 100% 的历史套餐类型
- 三类邮件触发场景的文案均已通过审核

## 3. 构建嵌入式 AI 助手知识库

**适用时机：** 你的产品有（或计划有）一个产品内置 AI 客服，需要一个 RAG 知识库——并且随着产品迭代，这个知识库需要保持准确和最新。

### 3.1 场景描述

你的 SaaS 产品新增了一个嵌入式 AI 助手，能在产品内回答用户问题（"怎么导出结果？"、"Pro 套餐包含什么功能？"、"这笔交易为什么被标记了？"）。AI 助手从你维护的结构化知识库中检索答案。

问题在于：这个知识库是上线时一次性写的，现在已经有三个月没更新了——定价过时、引用了已下线的功能、和当前帮助文档相互矛盾。PM 需要系统性地重建并维护它。

知识库需要包含以下模块：
- **用户手册** — 各功能的使用指引（分步说明，用户视角，无内部术语）
- **定价与套餐** — 当前套餐、账单 FAQ
- **常见问题** — 来自真实支持工单的 40+ 条 Q&A
- **API 参考** — 面向技术用户和集成场景的接口说明

### 3.2 起始提示词

```
/prd AI 助手知识库

我需要构建并维护嵌入式 AI 助手的 RAG 知识库。
结构以模块划分——每个 H2 标题是一个独立的检索单元。

需要的模块：
- user_manual/ — 功能使用指引（简洁英文，无内部术语）
- pricing/ — 当前套餐、账单 FAQ、付款方式
- faq/ — 约 40 条来自真实支持工单的 Q&A
- api_doc/ — REST/Webhook 参考、鉴权、速率限制、错误码

所有内容的硬性规则：
- 不能提及未上线的功能或尚未生效的定价
- 不能记录内部实现细节（评分算法、供应商名称、成本结构）
- 每一条声明都必须能追溯到 facts/ 或已批准的 PRD——不能凭空发挥
```

### 3.3 工作流

1. **AI** 读取 `facts/[产品].md` → 整理出当前已上线、已确认的功能清单
2. **PM + AI** 逐模块推进，从流量最高的模块开始（通常是 FAQ）
3. FAQ 模块：**PM** 粘贴一批真实支持工单；**AI** 提取 Q&A 对并格式化
4. **AI** 标记任何需要声明未确认功能的答案 → 标注 `[待核实]`
5. 定价模块：**AI** 读取 `facts/pricing.md` 并生成定价模块——不做任何发挥
6. 用户手册：**AI** 为每个功能生成分步指引，交叉引用对应的功能 PRD
7. **PM** 逐模块审核，检查准确性和语气
8. **AI** 提交到 `ai-kb/` 目录，每个模块附带 CHANGELOG 记录
9. **AI** 生成维护规则："每当 `facts/pricing.md` 更新，此模块必须同步更新"

### 3.4 知识库沉淀内容

- `ai-kb/`（或 `lumi-knowledge/`）— 已发布的 RAG 内容
- `memory/rules/rag-kb-authoring.md` — 撰写规范（"每个 H2 = 独立检索单元"、"不写内部实现"）
- `decisions/` — 哪些内容被排除在 AI 知识库之外以及原因（定价策略、内部算法、未上线功能）

### 3.5 完成标准

- 每一条内容都能追溯到权威来源（`facts/` 或已批准的 PRD）
- 撰写规范已保存到 `memory/rules/`，确保后续更新遵循同一标准
- AI 助手能正确回答一组 10 个测试问题，不出现幻觉

## 4. 追踪跨功能产品决策

**适用时机：** 多个功能共享同一底层策略（配额规则、权限、定价行为），但因为最初的决策原因没有记录下来，同一个问题被反复讨论。

### 4.1 场景描述

过去一年，你的 PM 团队交付了 6 个功能。其中 3 个涉及配额执行，2 个涉及套餐权限。但没人记得：

- 为什么免费套餐是每月 10 次查询（不是 5 次或 20 次）
- 为什么配额耗尽时是硬封锁（而不是软提示 + 自动升级）
- 为什么降级不立即生效，而是等到账单周期结束

每个迭代周期，工程师都会问"这个规则是什么"，PM 不得不从记忆或 Slack 历史记录中拼凑答案。

### 4.2 起始提示词

```
/prd 决策审计——配额与权限系统

我需要对配额和权限系统进行一次系统性决策审计。
对于以下 4 个问题，我需要正式的决策日志，记录决定内容、原因、假设条件和边界情况：

1. 免费套餐配额：为什么是每月 10 次，而不是其他数字
2. 配额耗尽时的硬封锁 vs 软提示 + 自动升级
3. 降级生效时间策略（立即生效 vs 账单周期结束）
4. 权限绑定：为什么功能访问权限绑定配额余额，而不只是套餐等级

在讨论每个问题之前，先检索 decisions/ 中已有的相关记录。
```

### 4.3 工作流

1. **AI** 扫描 `decisions/` 中所有涉及配额、权限或套餐逻辑的条目
2. **AI** 整理出哪些已有正式记录、哪些从未被记录
3. **PM + AI** 逐一讨论——PM 提供当初的决策背景；AI 提出追问
4. 针对每个问题，**AI** 起草 `decisions/decision-YYYYMMDD-[主题].md`，使用标准模板
5. **PM** 审核——调整假设条件和待定问题
6. **AI** 为相关决策添加交叉引用（"这个配额决策依赖与 decision-20250312-quota-exhaustion 相同的硬封锁逻辑"）
7. **AI** 生成 `memory/rules/quota-policy.md`——提炼成可复用的策略规则，供后续 PRD 自动加载
8. 提交

### 4.4 知识库沉淀内容

- `decisions/` — 4 条正式决策日志，包含原因、假设条件和边界情况
- `memory/rules/quota-policy.md` — 提炼后的策略规则，未来 PRD 自动加载
- 未来：任何功能涉及配额时，AI 自动加载这套规则作为上下文

### 4.5 完成标准

- 每个问题都有"为什么"的书面记录，而不只是"是什么"的答案
- 相关决策之间已建立交叉引用
- 下一位 PM 在 30 秒内能找到"为什么是每月 10 次"的答案

## 5. 数据驱动的流失原因诊断

**适用时机：** 发现某项指标下降（注册、激活、留存），需要综合多个数据源——埋点数据、支持工单、用户访谈——整理成有条理的假设和下一步行动。

### 5.1 场景描述

你注意到新免费用户 7 日留存率出现明显环比下降。手头有：

- **埋点数据** — 转化漏斗报告，显示用户在哪个步骤流失（注册 → 首次操作 → 第 3 天回访）
- **支持工单** — 三个月内被标记为"看不懂"或"不知道怎么用"的工单
- **用户访谈** — 5 位上个月流失的用户的访谈记录

你需要将这些数据综合成：流失根因排名、优先修复哪个的决策、以及一份可以分享给团队的研究总结。

### 5.2 起始提示词

```
/prd research 三月新用户流失原因分析

我需要诊断新免费用户为什么在第一周流失。

可用数据：
1. 埋点漏斗数据：从"首次操作"到"第 3 天回访"流失率 60%（我来粘贴数字）
2. 支持工单：3 个月内标记为"看不懂"的工单——我来粘贴 20 条样本
3. 访谈记录：上个月 5 位流失用户的访谈记录——我来粘贴

目标：列出 3 个主要根因并排名，选出优先修复项，
写一份可以分享给团队的研究总结。
```

### 5.3 工作流

1. **PM** 粘贴埋点数据；**AI** 识别漏斗中最陡峭的流失节点，建立初步假设
2. **PM** 粘贴支持工单样本；**AI** 按主题聚类（"找不到 X"、"搞不懂 Y"）并量化
3. **PM** 粘贴访谈记录；**AI** 提取直接引用，并将其映射到假设列表
4. **AI** 按证据权重对假设进行排名：定量信号 + 定性信号 + 用户引用
5. **PM + AI** 讨论排名第一的假设——排除干扰因素，确定最简单的验证方式
6. **AI** 写入 `memory/research/churn-new-users-march.md`——结构化研究总结
7. **AI** 生成 `decisions/decision-YYYYMMDD-churn-fix-priority.md`——记录为什么优先修复假设一而非假设二
8. 提交

### 5.4 知识库沉淀内容

- `memory/research/` — 结构化研究总结（假设、证据、置信度）
- `decisions/` — 优先级判断的原因（什么信号驱动了这个决策）
- 未来：处理新用户引导功能时，AI 自动加载这份研究作为上下文

### 5.5 完成标准

- 3 个假设各自有来自至少 2 个独立数据源的证据支撑
- 选定的修复方向有可证伪的成功指标（"3 周内 7 日留存率提升 X%"）
- 研究总结不超过 1 页——没有参与讨论的 PM 也能直接读懂
